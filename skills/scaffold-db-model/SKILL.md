# /scaffold-db-model — Generate a pygeia-compliant DB model

Generates a new SQLAlchemy DB model following `code_gen_db_models.md`. Produces the model file in the **flat** `db_models/` folder, the test file, and prepares the migration.

## Usage

- `/scaffold-db-model` — prompts for inputs.
- `/scaffold-db-model domain=continuous_care subdomain=clinical_notes entity=template scope=join:patient,PatientModel.partner_id` — non-interactive.

## Behavior

1. **Resolve pygeia** via `${GARAGE_ROOT}/scripts/resolve-pygeia.sh`. Halt on failure.
2. **Load the template:** `${PYGEIA_ROOT}/code_gen_db_models.md`.
3. **Load supporting standards:**
   - `${PYGEIA_ROOT}/docs/code-standards/db-models.md`
   - `${PYGEIA_ROOT}/docs/code-standards/partner-scope.md`
   - `${PYGEIA_ROOT}/docs/code-standards/database-migrations.md`
4. **Validate inputs:**
   - `domain`, `subdomain` (optional), `entity` — must be consistent with the existing entity (`entities/<subdomain>/<entity>.py` should already exist).
   - `scope` — one of:
     - `direct` → model has `partner_id` column directly.
     - `join:<rel>,<RelatedModel>.<col>` → model joins through a relationship.
     - `none` → truly reference data, no scope.
   - Refuse `none` if any field suggests relation to patient / customer / user / staff.
5. **Naming verification** per `subdomain-namespaces.md`:
   - File: `db_models/<subdomain>_<entity>_model.py` (FLAT — never under `db_models/<subdomain>/`).
   - Class: `<Subdomain><Entity>Model`.
   - `__tablename__`: `<subdomain>_<entities>` (plural entity).
6. **Generate the model.** Inherit BaseModel; rely on it for `created_at` / `updated_at` (do not redefine timestamps); define `partner_scope_path()` per the chosen scope:
   ```python
   @classmethod
   def partner_scope_path(cls):
       return cls.partner_id          # direct
       # or: return cls.<rel>, <RelatedModel>.partner_id   # join
       # or: return                    # none
   ```
7. **Generate the test file** at the mirrored test path.
8. **Generate the migration.** Run `make <domain>.db.gen.migration MESSAGE="add <subdomain>_<entity> table"`. Then **stop and ask** the user / orchestrator to review the generated migration before continuing — schema migrations are high-risk and the implementer must confirm reversibility, no destructive ops on populated tables, and FK cascade choices.
9. **Self-review:** flat path confirmed, BaseModel inheritance, no shadowed timestamps, `partner_scope_path()` present and matching the chosen scope, table name plural-entity, test file present.
10. **Report.** Files created + a one-line summary of the migration's `upgrade` / `downgrade` for human verification.

## Anti-patterns to refuse

- Placing the model in `db_models/<subdomain>/<entity>_model.py` (flat folder is required by template).
- Returning `None` from `partner_scope_path()` when the entity has a relation to a scoped entity.
- Redefining `created_at` / `updated_at` instead of inheriting.
- New FK without an explicit `ondelete` clause.
- Migrations that drop columns or add `nullable=False` without backfill, in production-relevant tables.

## Tools

- **Read** — template + standards.
- **Write** — model + test files. The migration file is generated by `make`; you only edit its contents after generation.
- **Edit** — update the generated migration's `upgrade` / `downgrade`.
- **Bash** — resolver, `make`, `git status`.
- **Grep, Glob** — verify domain layout.
