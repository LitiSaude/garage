Chapter 01 · Architecture

Architecture

One client · one app · one domain · one worker. The commerce process is the only thing that talks to Vindi and Xis. Everything else (continuous_care, customer_lifecycle) is read-only context for the order-create middleware.

High-level architecture

Client → FastAPI → domain + worker → lib adapters.

Routers stay single-domain — only import domains.commerce. Cross-domain hydration happens in middleware.

commerce_app · FastAPI
commerce_worker · Temporal

cross-domain RO reads
commerce_app + commerce_worker + lib/ adapters · cross-domain reads through middleware only
Why a middleware (not a router-level orchestrator)
Routers stay single-domain — only import domains.commerce. Cross-domain hydration (Customer, Patient, Prescription) lives in web_api/middlewares/commerce/, which the cross-domain-import CI script doesn't scan. The router only ever sees pre-hydrated objects on request.state.
Subdomain layout

Three subdomains v1 — and what stays out.

Inside commerce/: catalog (cross-channel), orders (channel-agnostic lifecycle), pharmacy (channel-specific extensions). Future channels plug in alongside pharmacy/ as new subdomains.

Extensibility

Add a pharmacy. Add a channel. Two different costs.

The whole point of the channel-discriminated catalog and the provider-agnostic webhook router is that v2/v3 are additive PRs — never reshapes.

Adding a new pharmacy

e.g. — Drogasil, Pague Menos, Onofre

One new adapter under lib/pharmacy_providers/adapters/<provider>/ implementing the abstract surface. One new PharmacyProvider enum value.

  • lib/pharmacy_providers/adapters/<provider>/adapter.py — implements abstract methods
  • lib/pharmacy_providers/adapters/<provider>/webhook_parser.py — verify + normalize
  • Add to PharmacyProvider enum
  • Add factory branch in lib/pharmacy_providers/factory.py
  • Optional: extend routing.select_provider if partner-restricted
What stays unchanged
The webhook router. The domain. The DB schema. All endpoints. POST /commerce/v1/webhooks/pharmacies/<new> works automatically.

Adding a new channel

e.g. — LAB_TEST, DEVICES, SUPPLEMENTS

One new row-class in products. One new seed script. One new sync routine. One new entry in quote_cart's dispatch table. New subdomain alongside pharmacy/.

  • Add value to OrderChannel enum (= products.channel)
  • lib/<channel>_providers/ — own adapter abstraction
  • domains/commerce/<channel>/ — channel-specific subdomain
  • private/scripts/commerce/seed_catalog_<channel>.py
  • sync_<channel>_catalog_routine.py
  • One entry in quote_cart's dispatch table
What stays unchanged
The internal data model + dispatch tables. v2 adds new public endpoints (e.g., /commerce/v1/products, /commerce/v1/cart/quote) — additively, never reshapes. /commerce/v1/orders already accepts items from any channel. The channel discriminator already exists.
