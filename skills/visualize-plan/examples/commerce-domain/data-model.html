Chapter 03 · Data model

Data model

Six new tables · two append-only at the DB level. order_status_events and pharmacy_webhook_events have UPDATE/DELETE revoked via Alembic migration on the application role. Six months later, any transition is traceable to the webhook (or workflow / cron) that caused it. The auxiliary medication_sku_mapping table joins continuous_care medications to commerce SKUs and powers the BFF lookup.

Entity-relationship

Five tables, color-coded by subdomain.

Cross-domain references (customer_id, patient_id, partner_id, prescription_id, prescribing_staff_id) are stored as plain UUIDs — never as FKs. Snapshot fields denormalize the related entity at order-create time so cross-domain deletes don't cascade.

Payment status, separately
payment_status is mutated only by Vindi-driven paths: charge_activity, the Vindi bill.paid webhook, and expire_stale_charges_routine. Sequence: PENDING → AWAITING_GATEWAY → PAID · or PENDING → FAILED · or HYDRATION_FAILED (S3 read exhausted) before any charge attempt.
